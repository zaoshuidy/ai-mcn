# 阶段验收标准（Acceptance Criteria）

## 0. 统一评分模型

所有阶段使用同一评分模型（机器可读版本见 `config/quality_gates.yaml`）：

| 评分维度 | 权重 |
| --- | --: |
| 业务价值与需求匹配 | 20 |
| 技术与执行可行性 | 15 |
| 输出质量 | 15 |
| 可复用性 | 10 |
| 第三方工具成熟度 | 10 |
| 风险与合规 | 10 |
| 可验证性 | 10 |
| 面试展示价值 | 10 |
| 总分 | 100 |

## 1. 统一门禁

```text
90—100分：允许进入下一阶段
85—89分：局部返工后重新评分
低于85分：重新设计或重新执行
存在真实性、密钥泄露、合规或核心功能问题：直接退回
```

> 评分与放行由项目总控执行；技术执行 AI 只提供证据，不自评通过。

## 2. 阶段清单

### Stage 0：项目控制台与组件准入机制

- **阶段目标**：建立可持续执行、可记录组件来源、可进行阶段验收的项目基础仓库。
- **输入**：阶段 0 任务说明书。
- **核心任务**：目录结构、项目文档、15 阶段验收门禁、组件准入机制、环境变量模板、日志框架、基础自动化检查。
- **交付物**：完整仓库骨架、`project/` 五份文档、`registry/` 六份准入文件、`config/` 两份配置、验证脚本、结构测试。
- **验收证据**：目录树、`validate_stage_0.py` 运行输出、pytest 运行输出、ruff 检查输出、git status 输出。
- **评分门槛**：≥90 分。
- **失败处理**：按缺失项补齐后重新运行验证脚本与测试。
- **是否允许进入下一阶段**：由项目总控根据评分决定。

### Stage 1：品牌 Brief 结构化

- **阶段目标**：将品牌方 Brief 转为结构化 YAML/JSON，作为全流程输入。
- **输入**：真实品牌 Brief（人工执行者提供）。
- **核心任务**：设计 Brief Schema；完成至少 1 份真实 Brief 结构化；覆盖产品、目标人群、卖点、禁忌、预算、交付要求。
- **交付物**：Brief Schema、结构化 Brief 文件、字段说明文档。
- **验收证据**：结构化文件通过 Schema 校验；人工执行者确认与原始 Brief 一致。
- **评分门槛**：≥90 分。
- **失败处理**：补充缺失字段或修正歧义字段后重新评审。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 2：达人搜索策略与候选采集

- **阶段目标**：制定可复用的达人搜索策略，采集候选达人池。
- **输入**：Stage 1 结构化 Brief。
- **核心任务**：关键词与类目策略；人工真实搜索；记录候选达人链接、截图、基础数据。
- **交付物**：搜索策略文档、候选达人清单（含链接与截图路径）。
- **验收证据**：每个候选均有真实小红书链接与截图；无虚构账号。
- **评分门槛**：≥90 分。
- **失败处理**：剔除无法验证的候选并补充采集。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 3：10 位达人筛选与评分

- **阶段目标**：从候选池筛选 10 位达人并给出量化评分。
- **输入**：Stage 2 候选达人清单。
- **核心任务**：建立达人评分维度（粉丝画像、互动质量、内容调性、商单适配、报价区间等）；逐位评分；输出排序。
- **交付物**：达人评分表、筛选理由文档。
- **验收证据**：评分依据可追溯至真实笔记与数据；评分表可复核。
- **评分门槛**：≥90 分。
- **失败处理**：重新校准评分维度或补采数据。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 4：最终达人证据与字幕语料

- **阶段目标**：锁定最终合作达人，补齐证据链与字幕语料。
- **输入**：Stage 3 评分结果。
- **核心任务**：人工确认达人真实性；采集代表视频字幕/口播文本；归档证据。
- **交付物**：最终达人清单、证据包（链接+截图）、字幕语料库。
- **验收证据**：每位达人至少 N 条语料（N 由阶段任务书确定）；语料来源可追溯。
- **评分门槛**：≥90 分。
- **失败处理**：更换达人或补采语料。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 5：达人风格画像与 Style Skill

- **阶段目标**：从语料提炼达人风格画像，沉淀为 Style Skill。
- **输入**：Stage 4 字幕语料库。
- **核心任务**：分析语言风格、结构、口头禅、节奏；形成风格画像文档；封装为可复用 Style Skill。
- **交付物**：风格画像文档、Style Skill 定义。
- **验收证据**：风格特征有语料例句支撑；Skill 可独立调用。
- **评分门槛**：≥90 分。
- **失败处理**：补充语料或修正画像。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 6：视频脚本 Skill 与工具选型

- **阶段目标**：完成脚本生成方向的三方向比较与工具/Skill 选型。
- **输入**：Stage 5 Style Skill、候选脚本工具与 Skill 清单。
- **核心任务**：调研候选方案；按组件准入机制评分；三方向比较；确定脚本生成方案。
- **交付物**：选型报告、组件评分卡、准入结论。
- **验收证据**：候选组件均完成准入登记与评分；选型理由明确。
- **评分门槛**：≥90 分。
- **失败处理**：补充候选或降低对单一组件依赖。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 7：三个脚本方向生成与竞争

- **阶段目标**：生成三个方向的脚本并竞争择优。
- **输入**：Brief、风格画像、Stage 6 选型结果。
- **核心任务**：按三个创意方向分别生成脚本；建立对比维度；总控评审择优。
- **交付物**：三份脚本、对比评分表、胜出脚本说明。
- **验收证据**：脚本完整可用；对比维度量化；评审记录完整。
- **评分门槛**：≥90 分。
- **失败处理**：按评审意见重写或补充方向。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 8：Humanizer 中文人味处理

- **阶段目标**：对胜出脚本进行人味化处理，去除 AI 痕迹。
- **输入**：Stage 7 胜出脚本。
- **核心任务**：检测并修复 AI 写作特征；保持达人风格一致性；前后对比。
- **交付物**：Humanizer 后脚本、修改对照表。
- **验收证据**：修改点逐条可解释；风格画像一致性校验通过。
- **评分门槛**：≥90 分。
- **失败处理**：回退过度修改并重新处理。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 9：分镜与拍摄可行性设计

- **阶段目标**：将脚本转为可执行分镜并评估拍摄可行性。
- **输入**：Stage 8 最终文案。
- **核心任务**：逐镜头分镜（画面、台词、时长、景别、道具）；拍摄可行性与成本评估。
- **交付物**：分镜表、拍摄可行性说明。
- **验收证据**：分镜与脚本一一对应；人工执行者确认可拍摄。
- **评分门槛**：≥90 分。
- **失败处理**：调整不可拍镜头。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 10：食品广告合规与事实质检

- **阶段目标**：对脚本与分镜进行食品广告合规与事实检查。
- **输入**：Stage 8 文案、Stage 9 分镜。
- **核心任务**：违禁词与绝对化用语检查；功效宣称审查；事实性断言核验；输出整改清单。
- **交付物**：合规检查报告、整改后定稿。
- **验收证据**：检查规则清单、命中项处理记录；无未处理的违禁项。
- **评分门槛**：≥90 分。
- **失败处理**：整改后复检；存在合规硬伤直接退回对应阶段。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 11：Agent 工作流编排

- **阶段目标**：将 Stage 1~10 能力编排为端到端 Agent 工作流。
- **输入**：各阶段产物与 Skill。
- **核心任务**：定义工作流节点与状态；实现编排代码；异常与人工介入节点设计。
- **交付物**：工作流代码、编排图、运行说明。
- **验收证据**：工作流可用测试输入跑通；节点日志完整。
- **评分门槛**：≥90 分。
- **失败处理**：修复断点并回归测试。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 12：飞书自动写入

- **阶段目标**：将交付物自动写入飞书文档。
- **输入**：工作流产物、飞书凭据（人工执行者配置）。
- **核心任务**：飞书 API 接入（组件准入后）；文档结构映射；写入与回读校验。
- **交付物**：飞书写入模块、写入样例文档链接。
- **验收证据**：真实写入成功截图与回读校验记录；凭据不入库。
- **评分门槛**：≥90 分。
- **失败处理**：降级为人工粘贴并记录原因。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 13：通用 MCN Skill 沉淀

- **阶段目标**：把项目流程抽象为可复用通用 MCN Skill。
- **输入**：全流程产物。
- **核心任务**：抽象输入输出契约；编写 Skill 文档；用新 Brief 验证可复用性。
- **交付物**：通用 Skill 包、复用验证记录。
- **验收证据**：Skill 可在新输入上运行；文档可被他人理解。
- **评分门槛**：≥90 分。
- **失败处理**：补充抽象层或文档。
- **是否允许进入下一阶段**：否（默认），总控评分后决定。

### Stage 14：GitHub 仓库与最终报告

- **阶段目标**：完成仓库整理与最终交付报告。
- **输入**：全部阶段产物。
- **核心任务**：仓库清理与文档完善；许可证与第三方声明核对；最终报告撰写；面试展示材料。
- **交付物**：最终仓库、最终报告、展示材料。
- **验收证据**：陌生人可按 README 复现；全部测试与验证脚本真实通过。
- **评分门槛**：≥90 分。
- **失败处理**：按验收意见补齐。
- **是否允许进入下一阶段**：项目结项评审。
